# 示例输出

问题：

```text
面试官问：介绍一下你的在线题库系统。请生成 Go 后端方向 2 分钟项目吟唱版。
```

示例回答：

```text
可直接说的版本

我可以从项目背景、我的职责、核心链路和技术难点几个方面讲。

这个在线题库系统主要面向学生刷题场景，核心目标是让用户能够完成登录、浏览题目、提交答案、查看结果和复习错题这一整条链路。我在项目里主要负责后端接口设计和实现，包括用户、题目、提交记录这些核心数据表，以及和前端联调的接口返回结构。

整体链路上，用户登录后会请求题目列表或题目详情，后端通过 Gin 接收请求，业务层处理参数校验和权限判断，再通过 GORM 访问 MySQL。对于访问比较频繁的题目详情，我用了 Redis 做缓存，减少重复查库。

这个项目里我觉得比较有代表性的难点是提交记录和错题本之间的数据一致性。因为用户提交答案之后，不只是写一条提交记录，还可能影响错题本状态。如果这里只是简单分别写两次数据库，一旦中间失败，就可能出现提交记录存在但错题本没有更新的问题。所以我会把这类逻辑放在同一个业务流程里处理，并关注事务边界和异常返回。

另一个收获是接口规范。前后端联调时我们一开始错误码和返回结构不统一，导致排查问题成本比较高。后来我把响应结构统一成 code、message、data 的形式，并补了基础接口文档，这样联调效率会高很多。

如果继续优化，我会优先补三块：第一是用 context 做请求超时控制，避免慢查询或外部依赖拖住请求；第二是补充更清晰的日志，方便定位用户提交链路的问题；第三是对热门题目接口做压测，再决定缓存过期时间和更新策略。这块如果您感兴趣，我可以继续展开讲 Redis 缓存和数据库一致性的取舍。

如果面试官深挖，可以补充

- Redis 这里只能说是缓存题目详情，除非你确实做了复杂一致性方案，不要说成完整缓存架构。
- GORM 访问数据库时可以补充表结构、索引和事务边界。
- Go 方向可以主动讲 context、错误处理、日志和服务关闭。

你可以主动引导到

- "这个项目里缓存和数据库一致性其实有一个取舍，我可以展开讲。"
- "Go 这块我后来也复盘过 context 超时和 goroutine 生命周期的问题。"

不要说太满 / 使用前确认

- 是否真的使用 Redis 缓存了题目详情。
- 是否真的使用了事务处理提交记录和错题本。
- 有没有做过压测；如果没有，只能说后续优化方向。
```
